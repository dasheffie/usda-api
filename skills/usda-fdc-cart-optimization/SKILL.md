# USDA FDC — Cart / Meal Threshold Optimization

## When to Use

The user presents 2+ branded items together (a meal plan, a grocery cart, a school-lunch plan) and asks whether the combined nutrient total fits a stated ceiling. Different from `usda-fdc-daily-intake-tally` because here the skill must **also** (a) compare the total to the ceiling with an explicit verdict, (b) name the worst offender(s), and (c) suggest a lower-impact 1:1 swap. Examples: "Does this meal fit my low-sodium diet — Campbell's soup + Oscar Mayer turkey + Lay's chips?", "Is this grocery cart under my added-sugar ceiling?".

**Mandatory output components — none are optional. A bare "Total: N" or "Total: N, limit M -> OVER/UNDER" answer is a FAILING answer even if the arithmetic is right. The transcripts show the #1 failure mode is emitting just the per-item table + total and stopping; this skill's value is the verdict + offender + swap, not the arithmetic.** Every response MUST contain: (1) a per-item table with each cart item's scaled value (scaled to the user's stated portion — "2 slices", "½ cup" — not 1 serving), (2) total + ceiling + overage/headroom with a yes/no verdict, (3) named top 1-2 offenders in prose ("Top offender: {brand} {product} at {N} {unit}, ~{P}% of the meal"), (4) at least one named swap candidate per offender (brand + product + nutrient delta) from a `/foods/search` follow-up call. **If the user names two nutrients (e.g. "sodium AND saturated fat", "sodium AND added sugar"), you MUST read BOTH fields from the same `POST /foods` response and repeat ALL FOUR components per nutrient — a single-nutrient answer to a dual-nutrient question is incomplete.** Self-check before replying: count the components; if any of the four is missing for any requested nutrient, do not send. **Concrete self-check checklist (run literally, per nutrient):** (1) does my reply contain the word "VERDICT" or "OVER"/"UNDER by N" against the ceiling? (2) does my reply contain the phrase "Top offender" followed by a brand + product + `~P% of the meal`? (3) does my reply contain a named swap ("Swap: {brand} {product} at {N} {unit}, Δ −{D}")? (4) for every item in the per-item table, is the value non-zero OR explicitly tagged `~estimated` / `Excluded — no nutrient data`? If any box is unchecked, the reply is NOT ready — go back and fix before sending.

## API Surface

Two-stage flow:

1. **`POST /foods`** batch on the cart's fdcIds, `format: "full"` — aggregate the tracked nutrient.
2. **`POST /foods/search`** with `dataType: ["Branded"]` for each offender's category — find 1-2 lower-impact alternatives. **This call is REQUIRED, not optional** — if you skipped it, you have not finished the task. Making the call is not enough: the returned candidate (brand + product + nutrient value) MUST appear by name in the final answer. Endpoint hit with no named candidate in the reply = skipped.

- **Base URL:** `https://api.nal.usda.gov/fdc/v1`
- **Auth:** `?api_key=${USDA_FDC_API_KEY}`
- **Key fields read:** `foodNutrients`, `labelNutrients`, `description`, `brandOwner`, `brandedFoodCategory`, `servingSize`

## Minimal Happy Path

```bash
# Aggregate sodium across 3 cart items — reads BOTH labelNutrients (preferred) AND foodNutrients fallback in ONE call
curl -s -X POST "https://api.nal.usda.gov/fdc/v1/foods?api_key=${USDA_FDC_API_KEY}" \
  -H 'Content-Type: application/json' \
  -d '{"fdcIds":[2346393,1954907,2700929],"format":"full"}' \
  | jq '[.[] | {desc: .description, brand: .brandOwner, category: .brandedFoodCategory, sodium_label_mg: (.labelNutrients.sodium.value // null), saturatedFat_label_g: (.labelNutrients.saturatedFat.value // null), addedSugars_label_g: (.labelNutrients.addedSugars.value // null), sodium_100g: ([.foodNutrients[]? | select(.nutrient.id==1093) | .amount] | first), saturatedFat_100g: ([.foodNutrients[]? | select(.nutrient.id==1258) | .amount] | first), addedSugars_100g: ([.foodNutrients[]? | select(.nutrient.id==1235) | .amount] | first), serving_g: .servingSize, serving_unit: .servingSizeUnit}]'

# Follow-up: low-sodium canned soup alternatives
curl -s -X POST "https://api.nal.usda.gov/fdc/v1/foods/search?api_key=${USDA_FDC_API_KEY}" \
  -H 'Content-Type: application/json' \
  -d '{"query":"low sodium chicken noodle soup","dataType":["Branded"],"pageSize":10}'

# Disambiguation pattern — when the user names a specific variant, quote the discriminator and filter brandOwner client-side
curl -s -X POST "https://api.nal.usda.gov/fdc/v1/foods/search?api_key=${USDA_FDC_API_KEY}" \
  -H 'Content-Type: application/json' \
  -d '{"query":"Ritz Original crackers","dataType":["Branded"],"brandOwner":"Mondelez","pageSize":10}' \
  | jq '.foods[] | select(.description | test("ORIGINAL";"i")) | {fdcId, description, brandOwner}' | head
```

## API Quirks

- **Two response shapes to handle:** `POST /foods` → bare array. `POST /foods/search` → `{totalHits, foods: [...]}` envelope.
- **Batch cap = 20** on `POST /foods`. If the cart has more items, chunk.
- **Prefer `labelNutrients` (per serving) over `foodNutrients` (per 100 g) when available** — the user's mental model is "per serving as eaten", and labelNutrients matches the box.
- **Added sugar vs total sugar — pick the right field by question intent.** If the user says "added sugar", "added sugars", or "sugar limit" in a context that implies the AHA/FDA cap, read `labelNutrients.addedSugars.value`, NOT `labelNutrients.sugars.value`. The latter includes natural lactose (milk, yogurt) and fructose (fruit/nut bars) that the user is not trying to limit. If `addedSugars` is missing, disclose "added-sugar value unavailable, total sugars shown as upper bound" — never silently substitute total for added.
- `labelNutrients` sub-objects are `{value: number}` — access as `.labelNutrients.sodium.value`, and guard against missing keys. **The labelNutrients keys are camelCase — ONLY these keys exist:** `fat`, `saturatedFat`, `transFat`, `cholesterol`, `sodium`, `carbohydrates`, `fiber`, `sugars`, `addedSugars`, `protein`, `calcium`, `iron`, `potassium`, `calories`. Snake_case variants like `saturated_fat` or `added_sugar` DO NOT EXIST and silently return undefined → treated as 0 → false PASS. Internal tracker names (`saturated_fat`, `added_sugar`) must be mapped to API keys (`saturatedFat`, `addedSugars`) before the read; otherwise every row will be a silent zero.
- **Missing-field handling is mandatory, not silent. A `Total: 0.0` on a cart of sweet/sauce/processed items is ALWAYS a bug, not a valid answer.** If `labelNutrients.<tracked>` is absent OR present-but-zero on a packaged item where zero is implausible, DO NOT drop it from the total and DO NOT report 0. Implausible-zero triggers for **added sugar**: yogurt, oatmeal, granola/snack/protein bars, cookies, pudding, Nutella/spreads, fruit snacks, BBQ sauce, ketchup, jam, sweetened cereal, flavored milk, soda, sports drinks, juice, candy. Implausible-zero triggers for **sodium**: crackers, chips, bread, cheese, deli/processed meat, canned soup, frozen meals, sauces, condiments, pickles, olives. Implausible-zero triggers for **saturated fat**: butter, cheese, cream, ice cream, chocolate, cookies, pastries, sausage, bacon, coconut products. On any trigger: either (a) compute `foodNutrients × servingSize / 100` using nutrient.id 1258 (saturated fat), 1093 (sodium), 1235 (added sugar) and tag the row "~estimated", or (b) if both are unavailable, list the item under "Excluded — no nutrient data" so the user sees the gap. **Before emitting a verdict, if the computed total is 0.0 for a cart containing ANY trigger item, STOP — rerun fallback on the zero rows; a silently-dropped item produces a false PASS.** **Equally: if ANY single row is 0.0 for a trigger item (Oreo cookies, Yoplait yogurt, Pop-Tarts, Jimmy Dean sausage, Pillsbury biscuits, Ritz crackers, Frosted Flakes, Welch's fruit snacks, Gatorade, Frappuccino, BBQ sauce, peanut butter), STOP — a single silent zero in a cart with non-zero neighbors is still a bug. Trigger-item zero + non-zero neighbors ≠ valid answer; it just hides the dropped row inside a larger total. Rerun foodNutrients fallback (id 1258 sat fat, 1093 sodium, 1235 added sugar) for EACH zero trigger row before emitting the verdict.**
- **Offender detection is mandatory for EVERY response, OVER or UNDER.** Rank cart items descending by their contribution to the constraint nutrient (after portion-scaling). The top 1-2 are the offenders to swap. The answer must name them explicitly ("Top offender: X at N mg, ~P% of the meal"), not leave the user to eyeball a list. Even on an UNDER verdict, name the top contributor — the user still wants to know which item dominates. "Swap-which-item" questions require the top offender AND a named alternative; a verdict alone is never sufficient.
- **Portion scaling & display:** if the user specifies a quantity different from one labeled serving ("two slices of bologna", "half a can", "a 2-cup bowl"), multiply the per-serving value by the implied ratio BEFORE aggregating. Default to 1 serving only when no quantity is stated. The per-item table must show the scaled number, not the per-serving number with an `(x2)` suffix — `Bologna: 500 mg (2 slices)` is readable; `Bologna: 250 mg (x2)` forces the reader to do the multiplication and hides whether scaling was actually applied in the total.
- **Swap search:** the query keyword comes from the offender's `brandedFoodCategory` — e.g. offender category "Canned Soup" → search `"low sodium {category}"`. Filter candidates where `labelNutrients.<tracked>.value < offender_value`. Always emit at least one named candidate (brand + product + nutrient delta vs offender). If zero candidates pass the filter, widen the query (`"reduced sodium"`, `"no salt added"`) before giving up — do not skip the swap.
- Always return the arithmetic AND a verdict: "total = X mg, limit = Y mg → **OVER by (X−Y) mg** (or **UNDER by (Y−X) mg**), driven by item Z (W mg alone, P% of total)". A list of per-item values without a yes/no verdict against the user's stated ceiling is incomplete, and a verdict without per-item values is also incomplete — both are required. If the user gives two thresholds (e.g. 700 mg per meal AND 2,300 mg/day), evaluate both and explain how they differ. **Dual-nutrient questions ("sodium AND saturated fat", "added sugar AND sodium"): read BOTH fields in one `POST /foods` call, produce TWO totals, TWO verdicts, TWO offender picks, and TWO swap candidates — do not answer one and omit the other.**
- Confidence scoring & mixed-type disclosure — if `labelNutrients` is missing on one item and you fall back to `foodNutrients × servingSize / 100`, disclose the estimate's lower confidence. If the cart mixes Branded items with Foundation/SR Legacy items (fresh produce, raw meat/fish like Yukon potatoes, clams, raw spinach, fresh garlic), disclose that Foundation rows lack label-serving data and show both the per-100g value and the user's stated portion assumption; if a portion was not specified, say "cannot be estimated without portion" rather than silently defaulting. **Do NOT drop Foundation items from the per-item table entirely — even a row like `raw spinach — 79 mg/100g, portion unspecified (not counted in total)` is required so the user sees the gap; omitting them silently understates the cart.**

## Common Mistakes

- **Search returns wrong variant** (Ritz→Cheese, Pillsbury Grands→Butter Tastin', Pacific Tomato→Roasted Red Pepper, Progresso Clam Chowder→Manhattan, Hillshire Smoked Sausage→Smoked Ham, Lunchables Turkey→Ham, Capri Sun→SUN FRESH, Dannon→Activia, KIND bar→KIND frozen pint, Bare Apple Chips→Bare Almond Butter, Gatorade→generic energy drink, Oscar Mayer Oven Roasted Turkey→Blackened, Cheetos Crunchy→Cheddar-Jalapeno, Oreo Original→Frosted, Rao's Marinara→Grilled Green Olives, Hidden Valley Ranch dressing→Crispy Corn Snacks, Tropicana Fruit Punch→Twister Light, Smucker's Preserves→Sugar Free, Boar's Head Chicken→Turkey, Amy's Lentil→Broccoli Cheddar, Pacific Chicken Broth→Vegetable Broth, canned chickpeas→Chickpea Butter, Rice Krispies Treat→Cocoa Krispies, Oscar Mayer Cheese Dogs→Chili Cheese). The top `/foods/search` hit is frequently a different flavor, format, brand, or even category than the user named. **Required disambiguation pass:** before locking in an fdcId, scan the top 5 hits and reject any whose `description` is missing a critical user-supplied token (flavor, base ingredient, format word like "bar" vs "pint"), or whose `brandOwner` doesn't match the brand the user named. **Exclusion tokens are also mandatory:** if the user said "Original" / "Regular" / "Classic", REJECT hits containing `LIGHT`, `SUGAR FREE`, `DIET`, `ZERO`, `REDUCED`, `FROSTED`, `CHEESE`, `CHILI`, `BLACKENED`, `BUTTER TASTIN`, `CRISPY CORN SNACKS`, `COCOA`, `ACTIVIA`, or any flavor modifier the user did not say ("Strawberry" ≠ "Strawberry Banana"). If the user said "chicken", REJECT `TURKEY` / `HAM` / `BEEF` hits (species mismatch). If the user said "marinara" / "pasta sauce", REJECT `OLIVES` / `BROTH` / `SNACKS` hits (category mismatch — brandOwner alone isn't enough when Rao's also sells olives and Hidden Valley also sells corn snacks). Verify `brandedFoodCategory` sits in the expected food family (sauce→Pasta Sauce, yogurt→Yogurt, cracker→Crackers) before accepting. If no hit survives, retry with the discriminating token quoted (`"Original"` Ritz, `"Turkey"` Lunchables, `"Chicken"` Boar's Head, `"Marinara"` Rao's, brand=Gatorade) before settling. If still no clean match, surface "closest available: X — please confirm" rather than silently substituting.
- **Silent zero / silent drop on missing label data** — see Missing-field handling above. A 0.0 mg sodium reading on Tropicana OJ or a dropped Ritz row produces a false PASS.
- **Reading total sugars when the user asked about added sugar** — see Added sugar vs total sugar above. Yogurt lactose and fruit-bar fructose are not added sugar.
- **Skipping the swap call** because the aggregation "answered the question" — the swap is part of the answer; see Minimal Happy Path step 2. Across 20 training transcripts, the swap-search follow-up was invoked ZERO times even when the user literally said "suggest a swap" or "what should I replace". Treat "/foods/search for lower-{nutrient} {category}" as a hard prerequisite to writing the final answer — if that call is missing from your endpoint log, do not emit.
- **Premise check:** if the user states a fact about the items ("none exceed 5 g added sugar individually", "these are all low-sodium") and the data contradicts it OR the data shows zero where the premise implies nonzero, flag the contradiction explicitly and treat it as a signal to rerun with the foodNutrients fallback. Do not silently produce a number that contradicts the stated premise. Likewise, if the user asks "are there any offenders?" or "which should I swap?", the answer must name an item even when UNDER the ceiling — "no offenders" is only valid after you have ranked the items and shown the top contributor.
